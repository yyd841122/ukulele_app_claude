# 开发流程文档 (DEVELOPMENT_WORKFLOW)

## 1. 角色与职责

### 1.1 角色定义

| 角色 | 主要职责 | 禁止行为 |
|------|----------|----------|
| ChatGPT (总指挥) | 技术架构、任务拆分、最终审核 | 不直接写业务代码 |
| Claude/MiniMax (执行 Agent) | 文档编写、代码实现、测试验证 | 不越权扩展功能 |

### 1.2 协作原则

1. **ChatGPT 是主脑**：所有技术决策、任务拆分、最终审核由 ChatGPT 负责
2. **Claude/MiniMax 是执行者**：按照 ChatGPT 的任务分配执行具体工作
3. **Agent 不能自主扩展范围**：所有功能必须来自 TODO 列表或 PM 明确需求
4. **异常必须停止**：遇到超出范围的问题，立即报告，不擅自决定

## 2. 任务执行流程

### 2.1 任务领取

```
ChatGPT 分配任务
  → Agent 接收任务（读取 TODO 文档）
  → Agent 确认输入/输出/验收标准
  → Agent 开始执行
```

### 2.2 任务执行

每个 Agent 执行任务时必须：

1. **读取相关文档**：PRD、MVP_SCOPE、当前 TODO
2. **确认范围**：检查是否在 MVP Scope 内
3. **执行实现**：按规范编写代码或文档
4. **自检**：对照验收标准检查
5. **输出报告**：填写 Task Report

### 2.3 任务交付

```
Agent 完成实现
  → 填写 Task Report
  → Chief Architect 审核
  → 通过：进入下一任务
  → 不通过：反馈修改意见，Agent 重新执行
```

## 3. 文档编写规范

### 3.1 文档必须自包含

- 每个文档必须包含完整上下文
- 不依赖"之前文档已说明"
- 新会话打开文档即可理解

### 3.2 禁止占位符

- 禁止使用"待完善"、"后续再补"等占位
- 如果确实不确定，标记为"[技术验证项]"并说明验证方式
- 所有章节必须有实质内容

### 3.3 中文为主

- 文档内容以简体中文为主
- 技术术语保留英文原文
- 如 Flutter、Dart、Riverpod、Drift 等

## 4. 代码编写规范

### 4.1 任务编码前检查

执行编码任务前，必须确认：

1. 该任务在 TODO_INDEX 中且状态为"可编码"
2. 有明确的输入/输出/验收标准
3. 架构文档已就绪
4. Flutter 项目已创建

### 4.2 编码约束

| 约束 | 说明 |
|------|------|
| Feature-First | 按 feature 组织代码 |
| Riverpod | 状态管理使用 Riverpod |
| Drift | 数据持久化使用 Drift |
| 无全局状态 | 避免全局 Provider 滥用 |
| 无过度设计 | MVP 允许简化 |

### 4.3 代码自检清单

完成代码后，自检：

- [ ] 代码能编译通过
- [ ] 符合 MVP_SCOPE 的功能范围
- [ ] 无硬编码字符串（使用 i18n 备选）
- [ ] 无 TODO 注释
- [ ] 单元测试覆盖核心逻辑

## 5. 测试规范

### 5.1 单元测试

| 类型 | 覆盖要求 |
|------|----------|
| Riverpod Provider | 必须有基础测试 |
| 数据模型 | 序列化/反序列化测试 |
| 业务逻辑 | 核心逻辑必须测试 |

### 5.2 Widget 测试

| 类型 | 覆盖要求 |
|------|----------|
| 独立组件 | 渲染测试 |
| 用户交互 | 基础交互测试 |

### 5.3 集成测试（后期）

| 类型 | 覆盖要求 |
|------|----------|
| 完整流程 | 关键路径 E2E |
| 离线场景 | 飞行模式测试 |

## 6. Git 工作流

### 6.1 提交约束

| 约束 | 说明 |
|------|------|
| 原子提交 | 一个 commit 做一件事 |
| 消息规范 | `type: description` 格式 |
| 不 push | MVP 阶段本地提交即可 |
| 测试通过后提交 | 确保不破坏构建 |

### 6.2 Commit Message 格式

```
type: description

types:
  docs: 文档修改
  feat: 新功能
  fix: Bug 修复
  refactor: 重构
  test: 测试相关
  chore: 构建/工具
```

### 6.3 分支策略

| 分支 | 说明 |
|------|------|
| main | 主分支，稳定可发布 |
| feature/xxx | 功能分支 |
| fix/xxx | 修复分支 |

MVP 阶段可简化为单分支，但保持 commit 原子性。

## 7. 异常处理

### 7.1 Agent 遇到异常时

```
发现异常
  → 判断是否为范围问题
    → 是：停止，报告 Chief Architect
    → 否：尝试解决
  → 解决失败：停止，报告
```

### 7.2 常见异常处理

| 异常 | 处理方式 |
|------|----------|
| 需求超出 MVP Scope | 报告 PM/Chief Architect |
| 技术方案不确定 | 报告并申请技术验证 |
| 依赖不可用 | 报告，寻求替代方案 |
| 测试失败 | 修复或报告 |
| 权限被拒绝 | 提供降级方案 |

## 8. 质量门禁

### 8.1 代码合并前检查

- [ ] 所有测试通过
- [ ] 无编译警告
- [ ] 无硬编码
- [ ] 符合 MVP Scope
- [ ] Chief Architect 审核通过

### 8.2 MVP 发布前检查

- [ ] 所有 P0 功能测试通过
- [ ] 离线流程完整
- [ ] 隐私政策就绪
- [ ] 权限声明完整
- [ ] 无崩溃日志
